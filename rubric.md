# Evaluation Rubric

This assessment is reviewed holistically. We are not looking for a perfect or fully production-ready solution. We are primarily evaluating engineering judgment, backend fundamentals, and effective use of AI-assisted development.

## What We Evaluate

### 1) Core Requirements Completion
We look at how well the submission covers the required scope, including:

- Authentication
- File upload
- Asynchronous processing
- Data persistence
- File retrieval endpoints
- Hosted UI
- Docker setup
- Documentation

A submission does not need to implement every detail perfectly, but it should clearly demonstrate a strong attempt at the core requirements.

### 2) Backend Engineering Fundamentals
We evaluate:

- API design and structure
- Separation of concerns
- Code organization
- Data modeling
- Handling of asynchronous workflows
- Maintainability and clarity of implementation

### 3) Architecture & Technical Judgment
We evaluate the candidate’s ability to make sound technical decisions, including:

- Framework and library choices
- Tradeoff awareness
- Simplifications made intentionally
- Queueing and processing design
- Hosting and deployment choices
- Reasoning documented in architecture and decisions files

### 4) Reliability & Error Handling
We look for evidence of thoughtful handling of failure cases, such as:

- Input validation
- Failed processing jobs
- Retry or fallback behavior
- Clear and structured error responses
- Logging and observability considerations

### 5) AI-Assisted Development Quality
AI use is encouraged. We evaluate:

- How clearly the candidate guided AI
- Whether the candidate validated AI-generated output
- Whether they improved or corrected weak AI suggestions
- Quality of prompts and workflows documented
- Evidence of engineering judgment rather than blind acceptance

### 6) Documentation Quality
We review the clarity and completeness of:

- `README.md`
- `ARCHITECTURE.md`
- `DECISIONS.md`
- `AI_USAGE.md`

### 7) Code Quality
We evaluate:

- Readability
- Consistency
- Naming
- Structure
- Simplicity
- General maintainability

### 8) Submission Quality
We also consider:

- Whether the backend is hosted and functional
- Whether the UI is hosted and functional
- Whether the repository is public and complete
- Whether instructions are clear enough for us to run or review the project easily

## Bonus Considerations

Bonus points may be given for thoughtful additions such as:

- Rate limiting
- Secure file handling
- Swagger / OpenAPI
- Tests
- Postman collection
- Monitoring or tracing plan
- Job monitoring visibility
- Strong structured logging
